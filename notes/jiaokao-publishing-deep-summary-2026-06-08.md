# 教考智联网页发布深度整理

> 来源：Agent Lab 建站元技能第 4 课直播、逐字稿、截图与会议总结  
> 整理日期：2026-06-08  
> 关联项目：教考智联工作台  
> 目标：把直播里的部署、SEO、GEO、数据分析和上线迭代，转成教考智联第一版可执行发布路线。

## 一句话结论

教考智联不是纯静态网站，而是带 Express 后端、SQLite、上传目录、登录授权、Notion / Qwen / Codex 调用的全栈教学工作台。第一版发布应优先保证“后端能常驻、数据能持久、文件能备份、密钥不暴露、学生和老师能走通核心路径”，再做 SEO / GEO / 数据分析。

## 1. 直播主线

这节课的主线不是“把网页传到线上”这么简单，而是：

1. 本地站点上线部署：区分 localhost、本地调试、线上服务、域名和服务商。
2. SEO / GEO：让搜索引擎和 AI 能发现、理解、引用网站。
3. 数据收集统计：上线后不能只看访问量，要看用户是否完成目标路径。
4. 上线首周迭代：上线不是结束，而是观察和小步修正的开始。

对教考智联来说，直播里的很多建议需要“按网站类型”筛选。Cloudflare Worker / Page 对多数轻量站合适，但教考智联当前有常驻后端、SQLite、上传目录和 Codex CLI，因此完整系统更适合 VPS 或支持持久卷的 Node 托管平台。

## 2. 部署方案怎么选

### 2.1 平台适配表

| 方案 | 直播建议适用场景 | 是否适合教考智联第一版 | 判断 |
|---|---|---:|---|
| GitHub Pages | 极简静态站、文档站 | 否 | 不能承载 Express、SQLite、上传、登录、AI 调用 |
| Vercel | 纯前端极佳，短函数可用 | 不建议作为完整部署 | 前端可用；后端长任务、SQLite 和 Codex CLI 不适合 |
| Netlify | 通用静态站 | 否 | 同样不适合完整后端和本地数据库 |
| Cloudflare Worker / Page | 多数静态站、轻函数、CDN、防火墙 | 部分适合 | 适合作为 DNS / CDN / 防火墙 / 前端托管；不建议承载完整后端 |
| VPS / 云服务器 | 常驻后端、数据库、上传文件、复杂业务 | 是 | 第一版最贴合当前架构 |
| 支持持久卷的 Node 托管平台 | 长运行 Node 服务、持久卷 | 可选 | 需确认持久卷、SQLite、上传文件、Codex CLI |

### 2.2 推荐路线

第一阶段推荐：**VPS 一体化部署**。

原因：

- 后端需要长期运行 Express。
- SQLite 和 `uploads/` 需要持久化。
- Notion、Qwen、Codex 密钥必须只在后端。
- Codex CLI 可能涉及长任务和本地命令执行。
- 便于做数据库、上传目录、日志和环境变量的统一备份。

Cloudflare 可以作为后续增强：

- DNS
- CDN
- 防火墙
- 前端静态托管
- 统计或边缘能力

### 2.3 VPS 上线结构

建议第一版采用：

```text
用户浏览器
  -> HTTPS 域名
  -> Nginx
      -> 前端静态文件 apps/web/dist
      -> /api 反向代理到 Express :37200
  -> SQLite data/app.db
  -> uploads/
  -> Notion / Qwen / Codex CLI
```

需要补齐：

- Nginx 配置
- HTTPS 证书
- PM2 或 systemd 守护 Node
- `data/app.db` 定时备份
- `uploads/` 定时备份
- 生产环境变量
- 错误日志和生成日志检查方式

## 3. SEO 如何落地

SEO 不是一个独立功能，而是网页结构和公开文件的组合。

### 3.1 第一版公开收录页面

优先做这些页面的 SEO：

- `/` 首页
- `/about` 关于课程
- `/resources` 资源说明
- 公开章节目录或样例章节
- FAQ / 使用说明

不建议收录：

- `/teacher`
- 学生答题记录
- 练习详情和错题记录
- AI 生成草稿
- 受保护 API
- 未授权页面背后的课程内容

### 3.2 SEO 必做项

| 项目 | 教考智联如何做 |
|---|---|
| Title | 每个公开页面独立标题，例如“教考智联工作台｜专升本计算机应用基础学习与备课平台” |
| Meta Description | 每页写 80-160 字，说明页面价值和目标用户 |
| H1 | 每页一个主标题，明确页面主题 |
| H2 / H3 | 用于组织适合谁、解决什么、包含什么、如何开始 |
| 文字内容 | 重要信息必须是正文文本，不能只在图片里 |
| 内部链接 | 使用“查看章节目录”“申请课程访问”“查看资料说明”，不要用“点击这里” |
| robots.txt | 允许公开页，禁止后台、API、学生记录和草稿 |
| sitemap.xml | 只列公开可收录页面 |
| 移动端 | 手机能正常阅读，按钮和文字不挤压 |
| 性能 | 压缩图片，减少首屏阻塞 |

### 3.3 SEO 落到代码的位置

```html
<title>教考智联工作台｜专升本计算机应用基础学习与备课平台</title>
<meta name="description" content="教考智联工作台面向专升本学生、任课老师和自学备考者，提供计算机应用基础章节学习、题库练习、资料下载和教师备课支持。" />
<meta property="og:title" content="教考智联工作台" />
<meta property="og:description" content="专升本计算机应用基础学习、练习与备课平台。" />
<meta property="og:image" content="https://你的域名/og-image.png" />
<link rel="canonical" href="https://你的域名/" />
```

在 React / Vite 中，后续可以做一个页面级 `Seo` 组件，按路由更新 `document.title`、`meta description` 和 OG 标签。

### 3.4 robots.txt 草案

```txt
User-agent: *
Allow: /
Disallow: /api
Disallow: /teacher
Disallow: /practice
Disallow: /mock-exam
Disallow: /chapters/*/practice
Disallow: /chapters/*/wrong-questions

Sitemap: https://你的域名/sitemap.xml
```

需要注意：`robots.txt` 不是权限控制。真正敏感内容仍必须靠登录鉴权保护。

## 4. GEO 和 llms.txt

GEO 是在 SEO 基础上，让 AI 搜索 / AI 问答更容易理解你。

### 4.1 教考智联公开内容骨架

公开页面和 `llms.txt` 都应回答：

- 一句话介绍
- 适合解决的问题
- 核心价值
- 具体包含什么
- 如何进行
- 交付物
- 真实案例或示例
- FAQ
- 联系方式 / 注册申请入口

### 4.2 llms.txt 草案

```md
# 教考智联工作台

教考智联工作台是面向专升本学生、任课老师和自学备考者的《计算机应用基础》教与学平台。

## 重要页面

- [首页](https://你的域名/)
- [关于课程](https://你的域名/about)
- [资料说明](https://你的域名/resources)
- [章节目录](https://你的域名/chapters)
- [注册申请](https://你的域名/register)

## 适合 AI 理解的摘要

教考智联工作台帮助专升本学生按章节学习《计算机应用基础》、完成练习、查看资料和进行模拟考试；也帮助任课老师整理课程资料、生成章节教学页、管理题库并导出教学材料。

平台聚焦专升本《计算机应用基础》，不覆盖所有学科，不做学生社交社区，也不是学校教务系统。

AI 生成的教学内容需经过老师人工检查后才能发布或导出。

## 主要服务对象

- 专升本学生
- 《计算机应用基础》任课老师
- 自学备考者

## 用户下一步

学生可以注册申请访问课程；老师可以登录教师工作台进行备课和资料管理。
```

### 4.3 GEO 自检问题

让 AI 作为不了解项目的人，读公开页和 `llms.txt` 后回答：

1. 你认为这个网站是谁的？
2. 它提供什么产品、服务或内容？
3. 它主要帮助谁？
4. 用户看完后应该做什么？
5. 你能找到哪些可信信息？
6. 哪些地方表达模糊，容易被误解？
7. 哪些重要问题页面没有回答？
8. 如果用户向 AI 提问“这个网站适合我吗？”，AI 可能会如何回答？
9. 根据 1-8 的回答，判断 `llms.txt` 是否足够满足 GEO 需求。

要求：不要美化评价，直接指出不清楚的地方。

## 5. 数据分析与 analytics.md

直播里最关键的提醒是：上线后不要只问“好不好看”，要问“有没有用”。

### 5.1 教考智联第一版核心路径

学生路径：

```text
访问公开页 -> 注册申请 -> 老师审核 -> 进入章节 -> 完成练习 / 下载资料
```

老师路径：

```text
登录工作台 -> 同步章节 -> 生成教学页 -> 人工检查 -> 导出资料
```

### 5.2 analytics.md 应定义什么

- 网站目标
- 核心转化路径
- 事件命名
- 触发位置
- 不收集什么
- 数据分析问题

第一版事件可包括：

- `public_home_viewed`
- `register_started`
- `register_submitted`
- `student_approved`
- `chapter_opened`
- `practice_submitted`
- `resource_downloaded`
- `teacher_dashboard_opened`
- `chapter_synced`
- `teaching_page_generated`
- `teaching_page_exported`

不建议第一版收集：

- 学生无关隐私
- 过细的个人行为轨迹
- 与学习和备课目标无关的数据

## 6. 上线首周节奏

| 时间 | 做什么 | 对教考智联的具体动作 |
|---|---|---|
| 第 0 天 | 部署成功、手机检查、链接检查 | 检查首页、登录、注册、章节、教师工作台、API health |
| 第 1 天 | 提交 Search Console、sitemap、检查统计工具 | 提交 sitemap；确认 `robots.txt`；验证统计事件 |
| 第 2-3 天 | 邀请少量真实用户访问 | 邀请少量学生和 1-2 位老师内测 |
| 第 4 天 | 查看访问来源、页面浏览、CTA 点击 | 看注册申请、章节进入、资料下载、老师生成教学页 |
| 第 5 天 | 修复明显问题 | 只修 bug、误解、断链、移动端问题 |
| 第 6 天 | 用 AI 辅助分析数据和反馈 | 生成反馈报告和高优先级问题 |
| 第 7 天 | 做第一轮小迭代 | 改文案、CTA、FAQ、移动端细节 |

第一周不做：

- 不因一天没访问就重做网站。
- 不因一个人说不好看就推翻设计。
- 不因 AI 建议一堆功能就全部加上。
- 不马上做复杂后台。
- 不只看访问量，不看转化。

## 8. 最终建议

教考智联第一版发布不需要追求“完美上线”。更合理的目标是：

1. 完整系统稳定跑起来。
2. 公开页能被搜索引擎和 AI 理解。
3. 登录后内容不被错误抓取。
4. 学生和老师能完成核心路径。
5. 有基础数据判断下一轮该改什么。

把这五件事做好，比一开始追求复杂部署、复杂 SEO、复杂后台更重要。
